# Feature Specification: Copilot Studio チャットで Adaptive Card 判断確定

**Feature Branch**: `001-confirm-adaptive-card`

**Created**: 2026-05-18

**Status**: Draft

**Input**: User description: "Copilot Studio チャットで Adaptive Card 判断確定"

## Clarifications

### Session 2026-05-18

- Q: 同一案件に対して複数判断者がほぼ同時に確定操作した場合、どの競合解決ルールを採用しますか？ → A: 先着確定優先（最初の成功のみ有効、後続は「既に確定済み」で失敗）
- Q: カード有効期限の扱いをどうしますか？ → A: 1回表示限り（再表示時は常に再発行）
- Q: 判断確定時に入力必須とする項目は何ですか？ → A: Code Apps と同じく、判断選択肢（承認・却下・差し戻し）と判断理由の2項目を必須にする
- Q: 判断確定後の結果共有チャネルはどれにしますか？ → A: 案件詳細画面 + 既存通知フローの両方
- Q: カードの確定操作を許可する対象ユーザー範囲はどれですか？ → A: 案件に割り当て済みの判断者のみ

## User Scenarios & Testing _(mandatory)_

### User Story 1 - チャット上で判断確定する (Priority: P1)

判断者は Copilot Studio チャット内で提示された判断候補を確認し、Adaptive Card の操作だけで判断を確定できる。

**Why this priority**: 既存の判断プロセスをチャットで完結させる中核価値であり、最小実用価値を構成するため。

**Independent Test**: 判断者が対象案件のカードを受け取り、カード内の確定操作を実行したとき、案件が判断済み状態になり結果が保持されることを単独で検証できる。

**Acceptance Scenarios**:

1. **Given** 判断待ち案件が存在し判断者がチャットを開いている, **When** 判断確定カードの確定操作を実行する, **Then** 案件は判断済みとして記録される。
2. **Given** 判断確定が完了した案件がある, **When** 同じカードで再度確定操作を行う, **Then** 重複確定は行われずユーザーに既処理であることが伝わる。

---

### User Story 2 - 不正・不適切な確定を防ぐ (Priority: P2)

判断権限のない利用者や対象外案件に対する確定操作は受け付けず、安全に拒否される。

**Why this priority**: 判断結果の信頼性と業務統制を守るため、誤操作防止よりも優先度が高い。

**Independent Test**: 判断権限のない利用者が同一カード操作を試行した際に、状態変更されず拒否メッセージが返ることを単独で検証できる。

**Acceptance Scenarios**:

1. **Given** 利用者が判断権限を持たない, **When** カードで確定操作を行う, **Then** 案件状態は変更されず権限不足が通知される。
2. **Given** カードの対象案件が既に無効または参照不可, **When** 確定操作を行う, **Then** システムは失敗理由を明示し不整合を残さない。

---

### User Story 3 - 確定結果を関係者へ共有する (Priority: P3)

判断確定後、申請者および関係者は確定結果を確認でき、必要な通知を受け取れる。

**Why this priority**: 判断確定の体験を完結させるために重要だが、確定そのものの成立より優先度は低い。

**Independent Test**: 判断確定後に申請者が結果表示または通知で内容を認識できることを単独で検証できる。

**Acceptance Scenarios**:

1. **Given** 判断が確定した, **When** 申請者が案件を確認する, **Then** 最新の判断結果と確定時刻を確認できる。
2. **Given** 判断が確定した, **When** 通知対象の関係者が通知を受ける, **Then** 同一案件で重複通知されず一貫した結果が共有される。

### Edge Cases

- カード操作時点で案件が削除済み、参照不可、または `Submitted` 以外だった場合はどう扱うか。
- カードは1回表示限りとし、同一案件を再表示する場合は常に新しいカードを再発行する。
- 同一案件に対して複数判断者がほぼ同時に確定操作した場合は、最初の成功のみ有効とし、後続操作は「既に確定済み」として拒否する。
- 判断確定時の入力必須項目は、判断選択肢（承認・却下・差し戻しの3択）と判断理由の2項目とする。
- カードの確定操作は、対象案件に割り当て済みの判断者のみ実行可能とし、未割り当ての判断者は拒否する。

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: システムは、判断待ち案件に対してチャット内で判断確定可能なカードを提示しなければならない。
- **FR-002**: システムは、カードからの確定操作を受けたとき、対象案件を一度だけ判断済みへ遷移させなければならない。
- **FR-003**: システムは、確定操作時に操作ユーザーの判断権限を検証し、権限不足時は状態変更を拒否しなければならない。
- **FR-004**: システムは、対象案件が無効・不存在・参照不可の場合、確定を中止し理由を利用者に返さなければならない。
- **FR-005**: システムは、判断確定時に判断結果、確定者、確定時刻を記録しなければならない。
- **FR-006**: システムは、判断確定後に申請者および対象関係者が結果を確認できる状態を提供しなければならない。
- **FR-007**: システムは、同一案件の重複通知または重複確定を防止し、競合時は最初の成功のみを有効として後続を「既に確定済み」で拒否しなければならない。
- **FR-008**: システムは、判断確定カードを1回表示限りとして扱い、同一案件を再表示する場合は新しいカードを再発行しなければならない。
- **FR-009**: システムは、判断確定時に Code Apps と同じ入力項目として、判断選択肢（承認・却下・差し戻しの3択）と判断理由の2項目を必須入力として受け付けなければならない。
- **FR-010**: システムは、判断確定後に案件詳細画面で最新結果を表示し、あわせて既存通知フローで対象関係者へ結果共有を行わなければならない。
- **FR-011**: システムは、カードの確定操作を対象案件に割り当て済みの判断者のみに許可し、未割り当てユーザーの操作を拒否しなければならない。
- **FR-012**: システムは、Code Apps と Copilot Studio のどちらから判断が作成されても、判断履歴を正本として案件状態・通知・表示結果を自動的に同じ結果へ整合させなければならない。

### Key Entities _(include if feature involves data)_

- **判断対象案件**: 申請内容、現在ステータス、判断結果、確定履歴を持つ業務対象。
- **判断確定操作**: チャットカードから送信される確定意図と、実行者・実行時刻・処理結果を表すイベント。
- **判断カード発行**: Adaptive Card の `cardInstanceId`、発行状態、消費状態、再発行による失効を追跡するレコード。
- **判断後整合処理**: 判断履歴の作成を契機に、案件状態更新と通知を自動的に実行する非同期処理。
- **通知対象関係者**: 判断結果を受け取る申請者・判断者・関係者の集合。

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 判断権限を持つ利用者の 95%以上が、チャットカード提示から 60 秒以内に判断確定を完了できる。
- **SC-002**: 正常な判断確定操作の 99%以上で、案件状態と判断履歴が不整合なく更新される。
- **SC-003**: 権限不足または無効案件への確定操作は 100% 状態変更されず、利用者へ理由が提示される。
- **SC-004**: 確定後 1 分以内に、対象関係者が結果を確認できる状態または通知を受け取れる割合が 95%以上となる。

## Assumptions

- 判断確定の実行権限は既存の判断者ロール定義をそのまま利用する。
- 案件ごとに「判断待ち」「判断済み」の状態概念は既存運用に存在する。
- 通知チャネル（チャット内表示または既存通知基盤）は既存仕組みを再利用する。
- 初期リリースでは 1 件ずつの個別判断確定を対象とし、カードからの一括確定は対象外とする。
- Code Apps と Copilot Studio はどちらも `ds_decision` 作成を判断確定の正本イベントとして扱い、案件状態更新は Power Automate の整合処理で自動反映する。
- Adaptive Card の発行状態は `ds_decisioncard` で追跡し、同一案件で再発行された場合は古いカードを失効させる。
